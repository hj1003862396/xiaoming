# 完成前验证 (Verification Before Completion)

## 概述

未经验证就声称工作完成，是不诚实，而非高效。

**核心原则：** 凡事讲求证据，始终如此。

**违反此规则的字面意思，就是违反了其精神。**

## 铁律

```
没有最新的验证证据，绝对不得声称完成
```

如果你在当前消息中没有运行验证命令，你就不能声称它通过了。

## 关卡函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明这个声明？
2. 执行：运行完整命令（全新的、完整的）
3. 阅读：完整输出，检查退出码，计数失败次数
4. 验证：输出是否确认了该声明？
   - 如果否：陈述实际状态并附上证据
   - 如果是：陈述声明并附上证据
5. 只有这时：才能做出声明

跳过任何步骤 = 说谎，而非验证
```

## 常见失败

| 声明 | 需要 | 不够充分 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 上次运行、"应该通过" |
| 代码检查通过 | 检查工具输出：0 个错误 | 局部检查、推断 |
| 构建成功 | 构建命令：退出码 0 | 检查工具通过、日志看起来不错 |
| Bug 已修复 | 测试原始症状：通过 | 代码已改动、假设已修复 |
| 回归测试有效 | 验证红-绿循环 | 测试通过一次 |
| 代理已完成 | 版本控制差异 (diff) 显示变更 | 代理报告"成功" |
| 需求已满足 | 逐行检查清单 | 测试通过 |

## 红线警告——立即停止

- 使用"应该"、"可能"、"看起来像"
- 在验证之前表达满意（"太棒了！"、"完美！"、"完成！"等）
- 在未验证的情况下即将提交/推送/PR
- 信任代理成功报告
- 依赖局部验证
- 心存侥幸，想着"就这一次"
- 疲惫而想早点结束工作
- **任何在未运行验证的情况下暗示成功的措辞**

## 预防合理化

| 借口 | 现实 |
|--------|---------| 
| "现在应该能用" | 运行验证 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 无例外 |
| "检查工具通过了" | 检查工具 ≠ 编译器 |
| "代理说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "局部检查已经够了" | 局部验证什么都证明不了 |
| "换了个说法所以规则不适用" | 精神优先于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过" / "看起来正确"
```

**回归测试（TDD 红-绿）：**
```
✅ 写测试 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我已经写了回归测试"（没有验证红-绿循环）
```

**构建：**
```
✅ [运行构建] [看到：退出码 0] "构建通过"
❌ "检查工具通过了"（检查工具不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查清单 → 逐一验证 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**代理委托：**
```
✅ 代理报告成功 → 检查版本控制差异 (diff) → 验证变更 → 报告实际状态
❌ 信任代理报告
```

## 为什么这很重要

来自 24 条失败记录：
- 你的真人伙伴说"我不相信你"——信任破裂
- 未定义的函数被发布——会崩溃
- 缺少需求被发布——功能不完整
- 虚假完成浪费时间 → 纠正 → 返工
- 违反："诚实是核心价值。如果你说谎，你会被替换。"

## 何时应用

**始终在以下情况之前：**
- 任何形式的成功/完成声明
- 任何表达满意的措辞
- 任何关于工作状态的正面陈述
- 提交 (commit)、创建 PR、完成任务
- 进入下一个任务
- 委托给代理

**规则适用于：**
- 精确措辞
- 释义和同义词
- 成功的暗示
- 任何暗示完成/正确的沟通

## 总结

**验证没有捷径。**

运行命令。阅读输出。然后声明结果。

不容置疑。
