# 测试反模式 (Testing Anti-Patterns)

**在以下情况加载此参考：** 编写或修改测试、添加模拟 (mock)，或想要向生产代码中添加仅测试用的方法时。

## 概述

测试必须验证真实行为，而非模拟行为。模拟 (mock) 是隔离的手段，而不是被测试的对象。

**核心原则：** 测试代码做什么，而非模拟 (mock) 做什么。

**严格遵循 TDD 可以防止这些反模式。**

## 铁律

```
1. 永远不要测试模拟行为
2. 永远不要向生产类添加仅测试用的方法
3. 永远不要在不理解依赖关系的情况下模拟
```

## 反模式 1：测试模拟行为

**违规：**
```typescript
// ❌ 差：测试模拟是否存在
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**为什么这是错误的：**
- 你在验证模拟有效，而非组件有效
- 模拟存在时测试通过，不存在时失败
- 对真实行为什么都说明不了

**你的真人伙伴的纠正：** "我们在测试一个模拟的行为吗？"

**修复：**
```typescript
// ✅ 好：测试真实组件或不模拟它
test('渲染侧边栏', () => {
  render(<Page />);  // 不模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 或者，如果侧边栏必须被模拟以进行隔离：
// 不要断言模拟——测试在侧边栏存在时 Page 的行为
```

### 关卡函数

```
在对任何模拟元素进行断言之前：
  问："我在测试真实组件行为还是只是模拟存在？"

  如果在测试模拟存在：
    停止——删除断言或取消对组件的模拟

  改为测试真实行为
```

## 反模式 2：生产代码中的仅测试方法

**违规：**
```typescript
// ❌ 差：destroy() 只在测试中使用
class Session {
  async destroy() {  // 看起来像是生产 API！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 清理
  }
}

// 在测试中
afterEach(() => session.destroy());
```

**为什么这是错误的：**
- 生产类被仅测试代码污染
- 如果在生产中意外调用，会很危险
- 违反 YAGNI 和关注点分离
- 将对象生命周期与实体生命周期混淆

**修复：**
```typescript
// ✅ 好：测试工具处理测试清理
// Session 没有 destroy() - 它在生产中是无状态的

// 在 test-utils/ 中
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 在测试中
afterEach(() => cleanupSession(session));
```

### 关卡函数

```
在向生产类添加任何方法之前：
  问："这只被测试使用吗？"

  如果是：
    停止——不要添加它
    改为放在测试工具中

  问："这个类拥有该资源的生命周期吗？"

  如果否：
    停止——这个类不适合这个方法
```

## 反模式 3：不理解就模拟

**违规：**
```typescript
// ❌ 差：模拟破坏了测试逻辑
test('检测重复服务器', () => {
  // 模拟阻止了测试所依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出——但不会！
});
```

**为什么这是错误的：**
- 被模拟的方法有测试所依赖的副作用（写配置）
- 为了"安全"而过度模拟破坏了实际行为
- 测试因错误原因通过或神秘地失败

**修复：**
```typescript
// ✅ 好：在正确层次模拟
test('检测重复服务器', () => {
  // 模拟慢的部分，保留测试需要的行为
  vi.mock('MCPServerManager'); // 只模拟慢的服务器启动

  await addServer(config);  // 配置已写入
  await addServer(config);  // 检测到重复 ✓
});
```

### 关卡函数

```
在模拟任何方法之前：
  停止——还不要模拟

  1. 问："真实方法有什么副作用？"
  2. 问："这个测试依赖于这些副作用中的任何一个吗？"
  3. 问："我完全理解这个测试需要什么吗？"

  如果依赖于副作用：
    在更低层次模拟（实际慢/外部操作）
    或使用保留必要行为的测试替身
    而不是测试所依赖的高层方法

  如果不确定测试依赖什么：
    首先用真实实现运行测试
    观察实际上需要发生什么
    然后在正确层次添加最小模拟

  红线警告：
    - "我会模拟这个以防万一"
    - "这可能很慢，最好模拟它"
    - 在不理解依赖链的情况下模拟
```

## 反模式 4：不完整的模拟

**违规：**
```typescript
// ❌ 差：部分模拟——只包含你认为需要的字段
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 缺少：下游代码使用的 metadata
};

// 稍后：当代码访问 response.metadata.requestId 时会失败
```

**为什么这是错误的：**
- **部分模拟隐藏结构假设** — 你只模拟了你知道的字段
- **下游代码可能依赖你未包含的字段** — 静默失败
- **测试通过但集成失败** — 模拟不完整，真实 API 完整
- **虚假的信心** — 测试对真实行为什么都证明不了

**铁律：** 模拟实际存在的完整数据结构，而非仅仅模拟你当前测试使用的字段。

**修复：**
```typescript
// ✅ 好：反映真实 API 的完整性
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 真实 API 返回的所有字段
};
```

### 关卡函数

```
在创建模拟响应之前：
  检查："真实 API 响应包含哪些字段？"

  行动：
    1. 从文档/示例检查实际 API 响应
    2. 包含系统可能在下游消费的所有字段
    3. 验证模拟完全匹配真实响应 schema

  关键：
    如果你在创建模拟，你必须理解整个结构
    当代码依赖省略的字段时，部分模拟会静默失败

  如果不确定：包含所有记录的字段
```

## 反模式 5：集成测试作为事后补充

**违规：**
```
✅ 实施完成
❌ 没有写测试
"准备好测试了"
```

**为什么这是错误的：**
- 测试是实施的一部分，而非可选的后续工作
- TDD 本可以防止这种情况
- 没有测试就不能声称完成

**修复：**
```
TDD 循环：
1. 写失败测试
2. 实施使其通过
3. 重构
4. 然后声称完成
```

## 当模拟变得太复杂时

**警告信号：**
- 模拟设置比测试逻辑更长
- 为了让测试通过而模拟一切
- 模拟缺少真实组件拥有的方法
- 模拟改变时测试失败

**你的真人伙伴的问题：** "我们需要在这里使用模拟吗？"

**考虑：** 使用真实组件的集成测试通常比复杂的模拟更简单

## TDD 防止这些反模式

**为什么 TDD 有帮助：**
1. **先写测试** → 迫使你思考你实际上在测试什么
2. **看它失败** → 确认测试在测试真实行为，而非模拟
3. **最少实施** → 没有仅测试方法悄悄进入
4. **真实依赖关系** → 你在模拟之前看到测试实际上需要什么

**如果你在测试模拟行为，你违反了 TDD** — 你在没有看到测试先于真实代码失败的情况下添加了模拟。

## 快速参考

| 反模式 | 修复 |
|--------------|-----|
| 断言模拟元素 | 测试真实组件或取消模拟 |
| 生产代码中仅测试方法 | 移到测试工具中 |
| 不理解就模拟 | 先理解依赖关系，最小化模拟 |
| 不完整模拟 | 完全反映真实 API |
| 测试作为事后补充 | TDD——先写测试 |
| 过于复杂的模拟 | 考虑集成测试 |

## 红线警告

- 断言检查 `*-mock` 测试 ID
- 方法只在测试文件中被调用
- 模拟设置占测试的 > 50%
- 删除模拟时测试失败
- 无法解释为什么需要模拟
- "以防万一"而模拟

## 总结

**模拟是隔离的工具，而非被测试的对象。**

如果 TDD 揭示你在测试模拟行为，你已经走错了。

修复：测试真实行为，或质疑你为什么要模拟。
