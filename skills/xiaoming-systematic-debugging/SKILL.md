# 系统性调试 (Systematic Debugging)

## 概述

随机修复浪费时间并引入新 bug。快速补丁掩盖了底层问题。

**核心原则：** 在尝试修复之前，始终先找到根本原因 (root cause)。症状修复就是失败。

**违反此流程的字面意思，就是违反了调试的精神。**

## 铁律

```
没有根本原因调查就不得修复
```

如果你没有完成第一阶段，你就不能提出修复方案。

## 使用时机

适用于任何技术问题：
- 测试失败
- 生产中的 bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况时使用：**
- 处于时间压力下（紧急情况使猜测变得诱人）
- "仅一个快速修复"看起来显而易见时
- 你已经尝试了多个修复方案
- 之前的修复没有生效
- 你不完全理解该问题

**以下情况不得跳过：**
- 问题看起来简单（简单的 bug 也有根本原因）
- 你很着急（赶工必然导致返工）
- 经理想要立即修复（系统性方法比反复试错更快）

## 四个阶段

在进入下一阶段之前，你必须完成每个阶段。

### 第一阶段：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整地阅读堆栈跟踪 (stack trace)
   - 记录行号、文件路径、错误码

2. **可靠地重现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果无法重现 → 收集更多数据，不要猜测

3. **检查近期变更**
   - 什么变更可能导致了这个问题？
   - Git diff、近期提交 (commit)
   - 新的依赖项、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断埋点：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传播
     - 检查每一层的状态

   运行一次以收集显示故障位置的证据
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的 Secrets: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量: ==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== 密钥链状态: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示了：** 哪一层失败（secrets → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误深在调用栈中时：**

   见此目录中的 `root-cause-tracing.md` 了解完整的向后追踪技术。

   **快速版本：**
   - 错误值从哪里产生？
   - 什么调用了传入错误值的这个？
   - 继续向上追踪，直到找到来源
   - 在来源处修复，而不是在症状处

### 第二阶段：模式分析

**在修复之前找到模式：**

1. **找到有效的示例**
   - 在同一代码库中找到类似的有效代码
   - 什么正常工作的东西与损坏的相似？

2. **与参考对比**
   - 如果实施某个模式，完整阅读参考实施
   - 不要略读——逐行阅读
   - 在应用之前完全理解该模式

3. **识别差异**
   - 有效和损坏之间有什么不同？
   - 列出每一个差异，无论多么小
   - 不要假设"那不会有影响"

4. **理解依赖关系**
   - 这需要哪些其他组件？
   - 什么设置、配置、环境？
   - 它做了哪些假设？

### 第三阶段：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 具体而非模糊

2. **最小化测试**
   - 做尽可能小的变更来测试假设
   - 每次一个变量
   - 不要同时修复多件事

3. **继续之前先验证**
   - 有效了吗？是 → 第四阶段
   - 没有效果？形成新的假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 研究更多

### 第四阶段：实施

**修复根本原因，而非症状：**

1. **创建失败测试用例**
   - 尽可能简单的重现
   - 如可能，自动化测试
   - 如果没有框架，用一次性测试脚本
   - 修复之前必须有
   - 使用 `xiaoming:xiaoming-brainstorming-test-driven-development` 技能 (skill) 编写正确的失败测试

2. **实施单一修复**
   - 解决已识别的根本原因
   - 每次一个变更
   - 不要"顺带"改善
   - 不要捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被破坏？
   - 问题真的解决了？

4. **如果修复无效**
   - 停止
   - 计数：你已经尝试了多少个修复？
   - 如果 < 3：返回第一阶段，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下面的步骤 5）**
   - 不要在未进行架构讨论的情况下尝试第 4 个修复

5. **如果 3 个以上修复失败：质疑架构**

   **指示架构问题的模式：**
   - 每个修复在不同位置揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"才能实施
   - 每个修复在其他地方创造新症状

   **停止并质疑基本原则：**
   - 这个模式从根本上是合理的吗？
   - 我们是否"凭借纯粹的惰性坚持下去"？
   - 我们应该重构架构还是继续修复症状？

   **在尝试更多修复之前与你的真人伙伴讨论**

   这不是一个失败的假设——这是一个错误的架构。

## 红线警告——停止并遵循流程

如果你发现自己这样想：
- "现在快速修复，以后再调查"
- "就试着改变 X，看看是否有效"
- "添加多个变更，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说 X，但我会以不同方式适配"
- "以下是主要问题：[没有调查就列出修复]"
- 在追踪数据流之前提出解决方案
- **"再试一个修复"（已经尝试了 2 个以上时）**
- **每个修复在不同位置揭示新问题**

**所有这些都意味着：停止。返回第一阶段。**

**如果 3 个以上修复失败：** 质疑架构（见第四阶段步骤 5）

## 你的真人伙伴发出的"你做错了"信号

**注意这些重定向：**
- "那没有发生吗？" — 你在未验证的情况下假设了
- "它会让我们看到……吗？" — 你应该已经添加了证据收集
- "别猜了" — 你在不理解的情况下提出修复
- "把这个超级想清楚" — 质疑基本原则，而非只是症状
- "我们卡住了？"（沮丧）— 你的方法行不通

**当你看到这些信号时：** 停止。返回第一阶段。

## 常见的合理化借口

| 借口 | 现实 |
|--------|---------| 
| "问题很简单，不需要流程" | 简单问题也有根本原因。流程对于简单 bug 来说很快。 |
| "紧急情况，没时间走流程" | 系统性调试比猜测-检查的反复试错更快。 |
| "先试试这个，然后再调查" | 第一个修复确立了模式。从一开始就做对。 |
| "确认修复有效后再写测试" | 未经测试的修复站不住脚。先测试才能证明。 |
| "一次修复多个节省时间" | 无法分离什么起了作用。会引入新 bug。 |
| "参考太长，我会适配模式" | 部分理解必然引入 bug。完整阅读它。 |
| "我看到了问题，让我修复" | 看到症状 ≠ 理解根本原因。 |
| "再试一个修复"（2 个以上失败后） | 3 个以上失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|-----------------|
| **1. 根本原因** | 读取错误，重现，检查变更，收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到有效示例，对比 | 识别差异 |
| **3. 假设** | 形成理论，最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试，修复，验证 | Bug 已解决，测试通过 |

## 当流程揭示"没有根本原因"时

如果系统性调查揭示问题确实是环境性的、时序依赖的或外部的：

1. 你已完成了流程
2. 记录你调查了什么
3. 实施适当的处理（重试、超时、错误信息）
4. 添加监控/日志记录以供将来调查

**但是：** 95% 的"没有根本原因"案例是调查不完整。

## 真实影响

来自调试会话：
- 系统性方法：15-30 分钟修复
- 随机修复方法：2-3 小时反复试错
- 一次修复成功率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
